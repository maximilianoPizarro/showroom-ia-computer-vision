= AI Gateway API key and curl inference

++++

NeuroFace chat, Lightspeed, and scaffolded apps reach MaaS through the **AI Gateway** on the hub — not the external MaaS endpoint directly. Obtain a Kuadrant API key and test inference with curl.

== AI Gateway configuration

image::m05-maas-inference.png[MaaS Inference Flow,align=center,width=100%]

[source,yaml]
----
# charts/all/developer-hub/files/software-templates/ai-computer-vision.yaml (excerpt)
maasEndpoint:
  default: https://ai-gateway.apps.__HUB_CLUSTER_DOMAIN__/v1
----

Platform services (Lightspeed, NeuroFace scaffold) use an auto-generated **platform API key** from Vault (`ai-gateway-platform-keys`). Workshop users request their own key through Developer Hub.

== Obtain your API key

. Open `https://developer-hub.%HUB_DOMAIN%/`
. Log in with `%USER_NAME%` / `Welcome123!`
. *Catalog* → System *workshop-kuadrant-apis* → API *MaaS LLM (ai-gateway)*
. *Kuadrant* tab → *Request API key* → copy the key

TIP: The gateway hostname is `https://ai-gateway.%HUB_DOMAIN%/v1/chat/completions`.

== Test chat completion with curl

[source,sh,role=execute]
----
export API_KEY="<your-kuadrant-api-key>"
curl -sk -H "Authorization: APIKEY ${API_KEY}" \
  -H "Content-Type: application/json" \
  -d '{"model":"llama-scout-17b","messages":[{"role":"user","content":"Explain PPE detection in one sentence"}]}' \
  "https://ai-gateway.%HUB_DOMAIN%/v1/chat/completions" | head -c 500
echo
----

.Expected output (truncated)
[source,json]
----
{
  "id": "chatcmpl-xxx",
  "object": "chat.completion",
  "choices": [{
    "message": {
      "role": "assistant",
      "content": "PPE detection uses computer vision..."
    }
  }]
}
----

== List available models

[source,sh,role=execute]
----
curl -sk -H "Authorization: APIKEY ${API_KEY}" \
  "https://ai-gateway.%HUB_DOMAIN%/v1/models" | python3 -m json.tool 2>/dev/null | head -20
----

== Alternative: OIDC instead of an API key

The `ai-maas` route accepts **either** credential — pick whichever is convenient. This is the same "dual auth" `AuthPolicy` pattern, just applied to the AI Gateway instead of a Keycloak-protected route:

[source,sh,role=execute]
----
hub-login %USER_NAME%
export CLIENT_ID="client-maas-%USER_NAME%"
export CLIENT_SECRET=$(oc get secret keycloak-client-maas-%USER_NAME% -n keycloak-system \
  -o jsonpath='{.data.clientSecret}' 2>/dev/null | base64 -d)
TOKEN=$(curl -sk -X POST "https://sso.%HUB_DOMAIN%/realms/maas/protocol/openid-connect/token" \
  -d "grant_type=client_credentials&client_id=${CLIENT_ID}&client_secret=${CLIENT_SECRET}" \
  | python3 -c "import sys,json; print(json.load(sys.stdin).get('access_token',''))")
curl -sk -H "Authorization: Bearer ${TOKEN}" -H "Content-Type: application/json" \
  -d '{"model":"llama-scout-17b","messages":[{"role":"user","content":"Hello"}]}' \
  "https://ai-gateway.%HUB_DOMAIN%/v1/chat/completions" | head -c 300
echo
----

In Developer Hub's Swagger UI (*Catalog* → *MaaS OpenAPI* → *Definition* tab), the *Authorize* dialog offers both **ApiKeyAuth** and **OAuth2ClientCredentials** — either one works on the same endpoint. API-key calls land on the `free`/`gold` rate-limit tier; OIDC calls land on a separate `oidc` tier.

== Contrast: direct MaaS (optional)

The hub OpenShift AI dashboard may still show the upstream MaaS endpoint for reference:

[source,text]
----
https://maas-rhdp.apps.maas.redhatworkshops.io/v1
----

Workshop apps in this pattern are wired to **AI Gateway** so Kuadrant can enforce APIKEY auth and rate limits. Do not embed upstream MaaS keys in scaffolded repos.

== What you learned

* All MaaS traffic from workshop apps flows through AI Gateway on the hub
* User API keys are issued via Developer Hub Kuadrant integration
* Platform keys (`ai-gateway-platform-keys`) are auto-generated in Vault at install time
* The `ai-maas` route accepts an API key **or** a realm-`maas` OIDC token on the same endpoint (dual auth)
