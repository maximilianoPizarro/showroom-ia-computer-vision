== Key design decisions

* RHCL Gateway API instead of OCP Routes — weighted multi-spoke load balancing
* Skupper instead of VPN — application-level cross-cluster connectivity
* Ambient Service Mesh — lower resource overhead on spokes
* Hub-centralized observability — Kiali, Thanos, OpenTelemetry on hub

== OIDC Integration

The pattern integrates Red Hat Build of Keycloak (RHBK) for OIDC authentication:

* **OIDC issuer** (RHCL OIDCPolicy): `https://sso.%HUB_DOMAIN%/realms/<realm>`
* **Admin / account consoles**: `https://keycloak.%HUB_DOMAIN%/` (realm `cv`, `backstage`, etc.)
* **Backstage realm**: Used by Developer Hub and DevSpaces login
* **cv, neuroface, maas realms**: Provisioned by the `rhbk-iam` chart for API protection
* **Workshop login**: `userN` / `Welcome123!`

The main NeuroFace route (`neuroface.%HUB_DOMAIN%`) allows unauthenticated access for EAST/WEST badges. The CV API route (`neuroface-cv.%HUB_DOMAIN%`) requires OIDC Bearer tokens from realm `cv`.

== Skupper Multi-Cluster Connectivity

Traffic flows from the hub to spoke clusters via Skupper service interconnect:

. Hub creates an `AccessGrant` (chart: `hub-interconnect`)
. A CronJob pushes `AccessToken` to each spoke via ACM `ManagedClusterAction`
. Each spoke creates a `Site` + `Connector` pointing to its local spoke gateway
. Hub `Listeners` create services that route to spoke connectors
. `HTTPRoute` backend weights are configured **per route** — for example `neuroface-app-lb` may use 50/50 east/west, while `neuroface-cv-lb` may steer 100% to one spoke depending on `values-hub.yaml`

The cluster badge in the NeuroFace UI header shows which spoke is serving:

* **EAST** (green badge)
* **WEST** (yellow badge)
* **HUB** (blue badge — fallback when API is unreachable)
