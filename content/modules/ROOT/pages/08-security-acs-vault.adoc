= Security — ACS & Vault
:_content-type: PROCEDURE

++++
<span class="workshop-time-badge">⏱ ~15 min</span>
++++

Security in this pattern spans two pillars: *Red Hat Advanced Cluster Security for Kubernetes (RHACS)* for vulnerability management and runtime protection, and *HashiCorp Vault* with *External Secrets Operator (ESO)* for secrets management.

== RHACS architecture

RHACS uses a hub-spoke model that mirrors the fleet topology:

[cols="1,2",options="header"]
|===
| Component | Cluster

| *Central* (control plane)
| Hub — hosts the RHACS dashboard, vulnerability database, and policy engine

| *Secured Cluster* (data plane)
| East + West — sensors collect vulnerability data and enforce runtime policies
|===

The RHACS Secured Cluster operator runs on each spoke and reports to Central on the hub. This gives you a unified security view across the entire fleet.

== Hands-on: open ACS Central

Navigate to the RHACS Central dashboard:

[source,sh]
----
https://central-stackrox.apps.{hub_domain}
----

Log in with your workshop credentials.

== Hands-on: explore the vulnerability dashboard

. Click *Vulnerability Management* in the left navigation
. Browse the list of discovered CVEs across all clusters
. Filter by cluster (`east`, `west`) to compare vulnerability posture
. Click on any image to see its full vulnerability report

[source,sh,role=execute]
----
hub-login user1
----

Check RHACS Central pods:

[source,sh,role=execute]
----
oc get pods -n stackrox -l app=central
----

== Hands-on: check secured clusters

Verify that both spokes are reporting to Central:

[source,sh,role=execute]
----
oc get securedclusters.platform.stackrox.io -n stackrox 2>/dev/null || echo "Check via ACS Central UI → Platform Configuration → Clusters"
----

In the ACS Central UI, navigate to *Platform Configuration* → *Clusters* and confirm both `east` and `west` appear as healthy secured clusters.

== Hands-on: review policy violations

. In ACS Central, click *Violations* in the left navigation
. Browse active violations — these are policy violations detected across the fleet
. Common violations include:
** Images with critical CVEs
** Containers running as root
** Deployments without resource limits

TIP: The pattern deploys with default ACS policies. You can create custom policies for your PPE workloads — for example, alerting when the NeuroFace model serving container uses an outdated base image.

== RHACS init-bundle synchronization

The pattern uses a GitOps-managed init-bundle to connect Secured Clusters to Central:

. The `acs-init-bundle-sync` chart on the hub generates the init-bundle secret
. ESO synchronizes the init-bundle to spoke clusters via Vault
. Secured Cluster operators on spokes use the init-bundle to establish trust with Central

This is fully automated — no manual bundle transfer needed.

== Vault & External Secrets Operator

HashiCorp Vault runs on the hub cluster and serves as the central secrets store:

[cols="1,3",options="header"]
|===
| Component | Description

| *Vault*
| Deployed in the `vault` namespace on the hub. Stores secrets (API keys, certificates, init-bundles).

| *External Secrets Operator (ESO)*
| Runs on hub and spokes. Syncs secrets from Vault into Kubernetes `Secret` objects.

| *SecretStore*
| ESO resource that connects to Vault using Kubernetes auth.

| *ExternalSecret*
| Declares which Vault paths map to which Kubernetes secrets.
|===

== Hands-on: explore Vault

Check Vault pods on the hub:

[source,sh,role=execute]
----
hub-login user1
----

[source,sh,role=execute]
----
oc get pods -n vault
----

Check Vault seal status:

[source,sh,role=execute]
----
oc exec -n vault vault-0 -- vault status 2>/dev/null | head -10
----

== Hands-on: check External Secrets

Verify ESO is running and secrets are synchronized:

[source,sh,role=execute]
----
oc get externalsecrets -A 2>/dev/null | head -20
----

Check the SecretStore configuration:

[source,sh,role=execute]
----
oc get secretstores -A 2>/dev/null | head -10
----

All ExternalSecrets should show `SecretSynced` status, meaning Vault secrets are successfully projected into Kubernetes.

IMPORTANT: Never expose Vault tokens or secret values in workshop environments. The pattern uses Kubernetes auth for Vault access — no static tokens are required.

== What you learned

* RHACS Central runs on the hub and collects vulnerability data from Secured Clusters on spokes
* The ACS dashboard provides a unified fleet-wide view of security posture and policy violations
* Vault stores all secrets centrally, and ESO synchronizes them to clusters as Kubernetes Secrets
* The RHACS init-bundle is distributed to spokes automatically via ESO and Vault
* GitOps manages the entire security configuration — no manual steps required
