= Cross-Cluster Networking with Skupper
:_content-type: PROCEDURE

++++
<span class="workshop-time-badge">⏱ ~10 min</span>
++++

Skupper Service Interconnect creates a secure application-layer network between clusters without VPNs, firewall rules, or special cloud networking. In this module you explore how spoke services become accessible on the hub through Skupper sites, connectors, and listeners.

== How Skupper works

Skupper establishes a *Virtual Application Network (VAN)* between clusters:

. A *Site* is created in a namespace on each participating cluster
. *Connectors* on spokes expose local services to the Skupper network
. *Listeners* on the hub create local `ExternalName` services that route traffic to the remote spoke connectors
. Traffic flows over mutual TLS (mTLS) tunnels — no direct API server or pod network connectivity required

[source,mermaid]
----
graph LR
    subgraph Hub
        L[Listener: yolo-ppe-serving] --> S1[Skupper Site]
    end
    subgraph East Spoke
        S2[Skupper Site] --> C1[Connector: yolo-ppe-serving]

    end
    subgraph West Spoke
        S3[Skupper Site] --> C2[Connector: yolo-ppe-serving]
        C2 --> K2[KServe Pod]
    end
    S1 ---|mTLS tunnel| S2
    S1 ---|mTLS tunnel| S3
----

== Resources in this pattern

The Skupper configuration is managed by the `spoke-interconnect` Helm chart:

[cols="1,2,2",options="header"]
|===
| Resource | Cluster | Purpose

| `Site`
| Hub + each spoke
| Establishes the Skupper site in the `service-interconnect` namespace

| `Connector`
| East, West
| Exposes `yolo-ppe-serving:8080` from `neuroface-cv` namespace to the VAN

| `Listener`
| Hub
| Creates a local service on the hub that routes to spoke connectors

| `AccessGrant` / `AccessToken`
| Hub → Spokes
| Secures the inter-site link with token-based authentication
|===

== Hands-on: inspect Skupper resources on a spoke

[source,sh,role=execute]
----
east-login user1
----

List all Skupper resources in the `service-interconnect` namespace:

[source,sh,role=execute]
----
oc get site,listener,connector -n service-interconnect
----

Inspect the Skupper site:

[source,sh,role=execute]
----
oc get site -n service-interconnect -o yaml
----

Check the connector that exposes the YOLO inference service:

[source,sh,role=execute]
----
oc get connector -n service-interconnect -o yaml
----

== Hands-on: inspect Skupper on the hub

[source,sh,role=execute]
----
hub-login user1
----

[source,sh,role=execute]
----
oc get site,listener,connector -n service-interconnect
----

The hub side should show *listeners* that bridge traffic to spoke connectors.

Check the Skupper pods:

[source,sh,role=execute]
----
oc get pods -n service-interconnect
----

== Hands-on: verify cross-cluster connectivity

From the hub, verify that the Skupper-bridged service is reachable:

[source,sh,role=execute]
----
hub-login user1
----

[source,sh,role=execute]
----
oc get services -n service-interconnect
----

You should see services created by Skupper listeners that forward to spoke connectors. These services make spoke-hosted KServe endpoints accessible from the hub as if they were local services.

== ExternalName services

Skupper creates `ExternalName` Kubernetes services on the hub that point to the Skupper router. This means hub workloads (like the NeuroFace gateway) can reference spoke services using standard Kubernetes DNS:

[source,text]
----
yolo-ppe-serving.service-interconnect.svc.cluster.local
----

The Gateway API HTTPRoute uses these services as backend references for the 50/50 east/west traffic split.

== What you learned

* Skupper creates a secure Virtual Application Network between clusters without VPNs
* Sites, connectors, and listeners are the three building blocks
* Spoke connectors expose KServe inference services to the network
* Hub listeners create local services that route traffic to remote spokes
* ExternalName services let hub workloads reference spoke services using standard Kubernetes DNS
